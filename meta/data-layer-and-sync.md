# Data Layer & Sync Architecture (Phase 3)

> How v0.5 Phase 3 *implements* the app-layer-ownership invariant from
> [auth-security-posture.md](auth-security-posture.md) / ADR 0008 §3, plus the delta-sync
> design. Formal contracts: ADR 0007 §4/§5. This is the reasoning behind the structure.

## Decisions
- **One owner-scoped repository is the only path to owned tables.** All reads/writes for the 8
  owned resources go through `app/repository.py::OwnedRepository`; routes never touch the
  session for owned models. Every query is born from a single `_owned()` helper that pins
  `user_id` to the caller and refuses any model not on an explicit allow-list.
- **Parent-FK ownership is checked centrally, not per route.** A `PARENTS` map declares every
  FK→owner relationship; inserts/upserts verify the caller owns each referenced parent before
  writing (e.g. a card's `subject_id`). Rejection is 422 and **deliberately indistinguishable
  from "parent does not exist"** — it never confirms whether a foreign id is real.
- **Upsert is keyed by the client UUID, with the existence check scoped to the caller.** An id
  owned by someone else is "not found" → falls through to insert → fails closed on the global
  PK as a 409. We do **not** use raw `INSERT … ON CONFLICT (id)` (it would silently overwrite a
  foreign row).
- **The delta cursor is a compound keyset `(updated_at|created_at, id)`, not a bare timestamp.**
  Pull always returns a resumable `next_cursor` **and** a `has_more` flag (not `null`-when-caught-up).
- **Soft-delete cascades `deleted_at` to sync-table descendants in app code.** A `CASCADE` map
  drives recursive tombstoning (subject → sources/cards/quizzes/grade-entries; quiz → questions).
  Append-only logs (`attempts`, `review_logs`) are never tombstoned; `cards` deliberately do **not**
  cascade from `sources`.
- **Endpoints are generated by a router factory** (`app/routers.py`) from a per-resource spec.
  Sync tables get full CRUD + batch upsert; append-only logs get insert + read only.
- **Storage is an injectable dependency** (`app/storage.py`), degrading to empty `storage_paths`
  when Supabase Storage is unconfigured. Generation persistence find-or-creates the subject by
  name; grading persistence is opt-in via an optional `question_id` on `/v1/grade`.

## Reasoning
- **Centralization is the security mechanism, not just DRY.** Because RLS does not guard the
  backend's owner-role queries, the invariant "every query is owner-scoped" can't rely on each
  route remembering a `WHERE user_id`. Funnelling through one helper (with the parent checks and
  cascade also centralized) makes the invariant *un-forgettable per resource* and gives the
  cross-user isolation tests a single surface to assault. A router factory reinforces this:
  uniform wiring means a new resource can't accidentally introduce an unscoped path.
- **Why the keyset cursor.** The backend mediates all writes through the ORM, and a *batch upsert
  commits in one transaction* → all rows share the exact same `updated_at` (`now()` is the
  transaction time). A bare-timestamp cursor with strict `>` would skip rows across a page
  boundary; with `>=` it would loop. Ordering by `(ts, id)` and seeking past the last `(ts, id)`
  is exact. This is a refinement of ADR 0007's "newest updated_at as the cursor," made necessary
  by batch writes — accepted cost: the cursor is opaque (base64), not a human-readable timestamp.
- **Why always-resumable `next_cursor` + `has_more`.** A client that has caught up still needs a
  position to poll for *future* changes. Returning `null` when caught up (the naive design) leaves
  it with nowhere to resume. So the cursor is the position after the last row (or the caller's
  echoed cursor when nothing changed), and `has_more` separately signals "fetch again now."
- **Why app-code cascade over a hard ORM/DB cascade.** A hard delete emits no per-child rows, so
  an offline client never learns its cached children are gone → ghost rows. Soft-deleting each
  descendant (bumping `updated_at`) makes every deletion surface as a tombstone on the next pull.
- **Why grading persistence is opt-in.** `/v1/grade` has legitimate ad-hoc use (grade without a
  stored question); requiring a `question_id` always would break that. Optional `question_id`
  keeps ad-hoc grading and persists an attempt only when the answer belongs to a stored question.

## Implications
- Adding a resource = add a model + a `ResourceSpec` + (if it has parents) a `PARENTS`/`CASCADE`
  entry. The ownership guarantees come for free; forgetting the `PARENTS` entry is the one thing
  to watch in review (it's the parent-stealing guard).
- `/v1/generate` and `/v1/grade` now **require a working DB** (they persist) — a behavior change
  from the stateless v0.4. Local dev without Postgres can no longer exercise them.
- Tests must run on **Postgres** (JSONB / checked-text enums / `timestamptz`); SQLite can't stand
  in. The suite builds schema from `Base.metadata`, not Alembic, to skip the Supabase `auth.users`
  FKs/RLS that need the `auth` schema — so tests don't prove the migration, only the ORM shape.
- **Pitfall captured:** never assign `func.now()` to an ORM attribute that will be serialized in a
  response — after commit (with `expire_on_commit=False`) the attribute holds the SQL clause, not
  a datetime. Rely on `SyncMixin.onupdate` to bump `updated_at`, and `refresh()` after a write to
  load server-computed values back for the response.

## Open questions
- Conflict policy edge: an upsert currently **resurrects** a soft-deleted row (last-writer-wins
  clears `deleted_at`). Confirm that's the desired offline-merge behavior when the iOS client lands.
- The opaque-cursor format is an undocumented server-internal contract; pin it (or version it)
  before the iOS sync client depends on it.
- Same FORCE-RLS / dedicated non-owner DB role question as in
  [auth-security-posture.md](auth-security-posture.md) — still open, revisit in Phase 4.
