# meta/ — Decision reasoning

The *why* behind how Cram is built — for a future developer or Claude session with zero context.
This complements (does not replace) the formal numbered decision record in
[`docs/adr/`](../docs/adr/): ADRs freeze the contracts/decisions; these files capture the reasoning,
trade-offs, and downstream constraints in one place.

| Topic | Summary |
|-------|---------|
| [auth-security-posture.md](auth-security-posture.md) | Fail-closed auth defaults, server-side JWT (ES256/JWKS; HS256 dev-only + JWKS required at boot; claim-presence enforced), app-layer ownership (RLS as defense-in-depth), minimal `/healthz`, deferred cost-DoS controls. A **2026-07-06 full-app pass** added a web HTTP-hardening layer (CSP/headers + login anti-enumeration — see web-dashboard.md). Cross-refs ADR 0005–0008. |
| [ios-auth-client.md](ios-auth-client.md) | **Client side of auth.** iOS signs in with the `supabase-swift` SDK (Keychain + auto-refresh kept as default), sends `Authorization: Bearer <jwt>` (drops `X-Cram-Secret`); token fetched per-request, never cached; auth gate only when configured; HTTPS-only bearer, enumeration-safe errors, local-scope sign-out fallback. Cross-refs ADR 0007/0008. |
| [data-layer-and-sync.md](data-layer-and-sync.md) | **Backend** Phase 3 structure: single owner-scoped repository, parent-steal/id-squat defenses, compound keyset delta cursor, app-code soft-delete cascade, router factory. Cross-refs ADR 0007/0008. |
| [exam-level.md](exam-level.md) | **Subject → Exam → Cards/Quiz (2026-07-07).** New owned+syncable `exams` table (migration `0005`, RLS + `auth.users` FK like every domain table) as a level between subject and card. `cards.exam_id`/`quizzes.exam_id` are **nullable `ON DELETE SET NULL`** — a card outlives its exam and falls back to the subject's **"General"** bucket (mirrors card→source); exam is deliberately **omitted from the soft-delete cascade** so deleting an exam never tombstones its cards. Optional owned-parent check on `exam_id`; `exam_date` per exam drives its countdown + SM-2 compression; `/v1/generate` gains an optional `exam_id`. Web adds a reusable a11y `Modal` + `ExamFormModal`/`SubjectFormModal` and fetches `exams` in the subject bundle. Contract mirrors a future iOS `Exam @Model`. **2026-07-07:** `subject.exam_date` retired (migration `0006`) now that exams own the date — web derives a subject countdown from its soonest upcoming exam; iOS reconciliation deferred to Phase 5. **2026-07-08:** grades link to exams via a nullable `grade_entries.exam_id` (`SET NULL`, migration `0007`, applied to prod); a graded exam is **derived-archived** into a web **"Past exams"** section (no status flag — graded ⇔ done, reversible) and its cards leave active revision; the Grades logger becomes a guided **Subject→Exam→grade** and "New subject" moves out of Grades; iOS gets `GradeEntry.examId` for sync parity (no UI). Cross-refs ADR 0007/0008. |
| [progress-model.md](progress-model.md) | **Practice vs assessment (2026-07-09).** The app's measurement contract: **Flashcards + Quizzes are practice** (record study time only — no SM-2, no attempts), and a **Review** is the sole assessment (recall phase → `PATCH /v1/cards` + `/v1/review-logs`; test phase → `POST /v1/attempts`). Those two writes are the only inputs to **readiness** = `0.5·cardMastery + 0.3·quizAccuracy + 0.2·coverage`, with a missing signal's weight redistributed and `untested` a first-class state (never a 0%). A `Question` has no SM-2 state and no FK to a `Card` (shared `topic` string only), so quiz answers deliberately **never touch a card's schedule** — the signals stay separate and combine only at the readiness layer. Practice writes no attempts (rather than tagged ones) because `attempts` has no source column and tagging needs a migration + iOS parity. One definition app-wide (`lib/readiness.ts`); `examReadiness()` falls back to the subject when an exam has no material of its own (the upload defaults to "General"). Cross-refs ADR 0002/0004, exam-level.md. |
| [read-path-performance.md](read-path-performance.md) | **Why prod felt slow, and what actually fixed it (2026-07-09).** Diagnosis first: Railway (`ams1`) and Supabase (`eu-central-2`) are colocated and `/healthz` TTFB is ~90 ms — **no hosting, region or database change was needed**. The cost was per-request and client-side: rendering the dashboard took **ten** HTTP requests (nine `loadDashboard` pulls + a tenth from `SidebarStreak`, which refetched review-logs on *every* page), nothing was cached, and each request paid a rate-limit Postgres write, a `pool_pre_ping` and a connection checkout. Fixes: a read-only **`GET /v1/dashboard`** aggregate (`repository.list_live`, owner-scoped, tombstone-free) + a one-entry client cache (`web/src/lib/api/cache.ts`; shares in-flight requests, 60 s TTL, evicts on rejection, cleared on sign-out) — every loader keeps its signature and slices the snapshot, so **no component changed**; app-level **gzip (16.8×)**; a bigger connection pool. **The per-resource delta endpoints are deliberately untouched — they are the iOS sync contract.** Hard constraint: **deploy the backend before the web.** Records the Starlette middleware-ordering trap that made gzip silently skip its own `minimum_size`. Cross-refs ADR 0009, cost-controls.md. |
| [ios-sync-client.md](ios-sync-client.md) | **Client side of sync (v0.5 Phase 5).** Offline-first, last-writer-wins, push-then-pull; sync metadata + `touch()`/`softDelete()` on the models; tombstone deletes; generated decks adopt server ids (de-dup); per-user cursors + user-switch wipe; trailing throttle + `Retry-After` 429 backoff. File-bytes not synced. Cross-refs ADR 0007/0008. |
| [cost-controls.md](cost-controls.md) | Phase 4: Postgres-backed per-caller rate limit + token-based per-user/global daily spend cap; why metering commits before persistence (a spend-cap-bypass fix); why the rate limit guards the DB not the wallet (size it for the sync client); generate degrades (keeps the deck) on a Storage hiccup; prod-config guard; **`rate_limit_buckets` lazy in-process prune (2026-07-06)**. **2026-07-09: the limiter is split by HTTP method** — writes + the paid AI endpoints keep the Postgres counter (it guards real money); safe `GET`/`HEAD` move to a separate, higher **in-process** ceiling (`CRAM_READ_RATE_LIMIT_PER_MIN`, default 240) with **no DB write**, because charging every read an `INSERT`+`COMMIT` on one hot bucket row serialised a user's concurrent reads on its lock. Consequence: `CRAM_RATE_LIMIT_PER_MIN` is now a *write* ceiling, so an under-sized value (the old ~10) bites harder — a review writes 2 rows per card. Cross-refs ADR 0009. |
| [landing-page.md](landing-page.md) | **Public marketing page (2026-07-09).** The owner's standalone HTML comp ported to `/home` (with `/` → `redirect("/home")`) as a **verbatim CSS Module**, not a Tailwind rewrite, so the reference cascade survives and the pixel diff is zero by construction; hashed class names prevent collisions, bare selectors are prefixed `.page`, and low-specificity resets use `:where(.page)` so they don't outweigh single-class button rules. Documents the three `globals.css`/preflight rules that silently move any foreign comp (`text-wrap: balance`, `html{line-height:1.5}`, zeroed `p` margins — the last one is load-bearing inside flex columns). **Reverses web-dashboard.md's "no `next/font`, system stack"**: the 2026-07-06 CSP (`font-src 'self'`) blocks the comp's Google Fonts link, so Geist is self-hosted via `next/font` — landing-only. Adds a real `/signup` route (`LoginForm` gains `initialMode`; the tab toggle uses `history.replaceState` so the anti-enumeration notice survives). Light-only canvas pinned via `html:has(> body > .page)`. Verified with a dependency-free CDP screenshot harness (offsets + total height exact; pixel-identical below the header). A **2026-07-10 mobile pass** scopes the verbatim-port doctrine to **desktop only** — the comp's own mobile CSS was broken, so below 1041px the page is authored, not ported. Fixes three latent bugs that `overflow-x: clip` had been hiding: a bare `1fr` track takes a **min-content floor from every item**, so the dashboard mock's `width: 235px` search bar forced the hero column to **421px inside a 358px container** and silently sliced off the headline + both CTAs (→ `minmax(0, 1fr)` everywhere); the hamburger was never a menu (`aria-label="Jump to features"`) while the nav links sat `display: none`; and `.storySteps { display: none }` deleted all four "How it works" steps, freezing the stage on scene 0. Now: a real drawer (`inert`, scroll-lock, Escape + focus return) that must be a **sibling of `<header>`, because the header's `backdrop-filter` makes it the containing block for `position: fixed` descendants** and collapses a nested scrim to zero height; a stacked story where each step renders its own scene inline (scenes authored once, rendered into both trees, one `display: none`); and a devices section **scaled** as a fixed 900×600 canvas via `tan(atan2())` (CSS length division) rather than reflowed, since the mockups' inner UI is absolute px and the comp hardcoded a 760px iPad into a 390px viewport. Touch targets keyed to `@media (pointer: coarse)`, not width. **Records that `/impressum` + `/datenschutz` still do not exist** (deferred) and what the disclosure surface already is. |
| [mascot.md](mascot.md) | **Interactive Claude pixel mascot (2026-07-09).** Why the sprites in `web/public/claude-mascot/` are **build output** of `web/scripts/build-mascot-sprites.py` and must never be hand-edited: the three owner-supplied packs disagree (1254px vs 512px canvases, two arm poses, the eight directional frames rendered at **+1.34°**, transparent pixels stored as *white* so any non-premultiplied resample halos, and a "generating" pack whose transparency checkerboard is **painted opaque pixels**). The script de-rotates, normalises on head-width + head-centre-x + feet-baseline, and emits light+dark. **Speech-bubble text is HTML, never a sprite** — a baked capital is 3px of 360, i.e. **1.2px** at an 84px mascot, needing a 471px mascot to reach a legible 7px; a resolution limit no dark variant can fix. Dark variants exist for the 34 bubble frames only, and classify black-on-orange as *eyes* vs black-next-to-white as *ink* so an invert can't whiten his eyes; the work frames are excluded because their white rectangles are the **paper cards he produces**, not bubbles. `theme` is an explicit prop because `landing.module.css` has no dark rules while `layout.tsx` stamps `.dark` from the OS — `auto` there put him asleep on a white page. He disengages from a stationary cursor (5s), runs weighted idle bits, sleeps at 30s (and in dark mode, waking on approach — dark mode is not night, so a literal reading would disable tracking for permanent-dark users), and plays a seated laptop loop while `mood="reading"`. **Rejected the `KebeliSamet0/clawd` artwork**: its `assets/LICENSE-clawd` carves the art out of the repo's MIT licence (All-Rights-Reserved, non-commercial, and the character is Anthropic's property) and it's a visibly different character — only its MIT *code* timings are reused. Cross-refs landing-page.md, web-dashboard.md. |

A **2026-07-02 visual redesign** adds per-subject color identity (deterministic, CSS-variable driven so it survives Tailwind JIT) + a `motion` interaction layer (sliding tabs, spring entrances), with `PRODUCT.md`/`DESIGN.md` as the design source of truth. A **2026-07-03 pass** adds full **light/dark theming** (semantic RGB-triple token layer, `darkMode: class`, no-flash toggle) and a **Linear-inspired restraint retune** (neutral surfaces, per-subject color demoted to one quiet tile, calmer state-driven motion; entrances no longer gate visibility). Forced the backend's env-driven `CRAM_CORS_ORIGINS` allowlist (default closed, credentials off).
