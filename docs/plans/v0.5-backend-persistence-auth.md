## Security + code review (2026-06-18)

A security review and a code review were run over the v0.5 backend (`972aec6` + the live schema).
**No Critical/High issues; the JWT verification is sound** (mutually-exclusive JWKS-RS256/ES256 vs
HS256 with algorithms pinned, audience + issuer + exp enforced; `.env` untracked). Items below.

**Fixed this session (`fix(backend): …`):**
- **S-M1 — dev fallback now fails closed.** The unauthenticated loopback→dev-user path is gated
  behind a new opt-in `CRAM_ALLOW_DEV_FALLBACK` (default off) and refused at startup in prod, so a
  deploy that forgets `CRAM_ENV=prod` no longer fails open behind a same-host reverse proxy.
- **S-add — JWKS errors return a clean 401, not a 500.** `_verify_token` now catches
  `PyJWKClientError` (bad `kid`, malformed/unreachable JWKS), logs server-side, masks the client
  response — no stack-trace leak, no auth outage masquerading as 500.
- **Cfg — `SUPABASE_JWKS_URL` was the publishable key, not a URL.** Would have failed *every* token
  in Phase 3. Corrected to the real endpoint; verified the project signs with **ES256** (asymmetric,
  JWKS) — the legacy `SUPABASE_JWT_SECRET` is unused. *(.env only; not committed.)*
- **S-M3 (partial) — uploads read in 1 MiB chunks with early abort**, bounding memory to
  ~`max_file_bytes`/file so a chunked-encoding body can't bypass the cap. (A hard reverse-proxy
  body cap is still wanted — see Phase 4.)
- **C-R1 — single cached `get_settings()`**; `auth.py`/`db.py`/`main.py` no longer each re-read env.
- **db hardening** — `normalize_url` now strips provider-only query params (`?pgbouncer=true`) that
  psycopg rejects (this bit us during setup).
- **C docs** — model comments added for the soft-delete-vs-hard-cascade trap (C-D1) and the
  `graded_at`/`reviewed_at` vs `created_at` event-time distinction (C-R2).

**Deferred — fold into the phases below (not bodged now):**
- **S-O1 / S-M2 → Phase 4:** no rate limit + no Anthropic spend cap (auth-gating shrinks it, but an
  authenticated user can still drive cost). A real spend cap needs the usage-tracking that lands
  with persistence — do it properly in Phase 4, not as an in-memory half-measure.
- **S-M3 (remainder) → Phase 4:** documented hard reverse-proxy body cap in front of the app cap.
- **C-D1 → Phase 3 (must internalize):** RLS does **not** guard the backend's own queries (the
  service connects as the table-owner role, `ENABLE` not `FORCE` RLS) — so **all** ownership
  enforcement is app-layer. Route every query through one owner-scoped helper and assert cross-user
  isolation in tests. Soft-delete must set `deleted_at` on descendants (a hard ORM cascade emits no
  child tombstones → ghost rows on offline clients).
- Rotate the DB password exposed in chat during setup (Supabase → Settings → Database).

## Goal

Turn the stateless v0.4 backend into the system of record: the full data model in Supabase
Postgres, per-user accounts via Supabase Auth, uploaded files in Supabase Storage, and a sync
contract the iOS app targets later. Ship the carried pre-deploy hardening. **All Windows-doable**
except the iOS sync client (Phase 5, Mac).

- **Out (deferred):** the iOS sync client (Mac/Xcode, Phase 5 below), the v0.4 iOS UI work, the web

## Phases

### Phase 0 — DB plumbing & config ✅ (done, `972aec6`)
- Add deps: `sqlalchemy`, `alembic`, `psycopg[binary]`, `pyjwt[crypto]`.
- New env vars in `config.py` (`Settings`): `database_url`, `supabase_url`, `supabase_jwt_secret`
  / `supabase_jwks_url`, `supabase_service_role_key`, `supabase_storage_bucket`.
- `app/db.py`: SQLAlchemy engine + session factory + FastAPI session dependency. Pooler-safe engine
  options; migrations use the direct URL.
- Alembic skeleton (`backend/alembic/`, `alembic.ini`), env wired to `Settings.database_url`.
- `/healthz` gains a DB connectivity check (`SELECT 1`).
- **Acceptance:** app boots, `/healthz` reports DB reachable, `alembic upgrade head` runs clean
  (empty migration).

### Phase 1 — Data model + initial migration ✅ (models `972aec6`; live migration run + verified 2026-06-18)
- SQLAlchemy models for all tables in ADR 0007 §4 (`app/models/`), UUID PKs, `created_at`,
  `updated_at`, `deleted_at` where specified; enums matching existing iOS/contract values.
- First Alembic migration creating every table + indexes (`user_id`, `subject_id`, `updated_at`).
- RLS policies as a migration step (SQL): enable RLS, `auth.uid()` policies — defense-in-depth.
- **Acceptance:** `alembic upgrade head` builds the full schema on the Supabase Postgres; downgrade
  works; a smoke test inserts/reads one row per table via the ORM.

### Phase 2 — Auth (backend-mediated JWT) ✅ (done, `972aec6`)
- `app/auth.py`: `get_current_user` dependency — verify Supabase JWT (JWKS or shared secret),
  extract `sub`, return a `CurrentUser`. 401 on missing/invalid/expired.
- Apply it to data endpoints; move `/v1/generate` and `/v1/grade` under JWT (ADR 0007 §3). Retire
  `X-Cram-Secret` from the client path (optionally keep an internal secret for system calls).
- Keep the `CRAM_ENV=prod` fail-closed guard; extend it to require auth config in prod.
- **Acceptance:** a valid Supabase test JWT is accepted and scopes queries to that user; a missing/
  forged/expired token is rejected 401; cross-user access is impossible (ownership-scoped queries).

### Phase 3 — CRUD + sync API + persisting AI output ✅ (implemented; tests green vs disposable Postgres)
- REST endpoints per resource (subjects, sources, cards, quizzes, questions, attempts,
  grade_entries, review_logs), all ownership-scoped (`user_id == current_user`).
- Sync primitives (ADR 0007 §5): `GET …?since=<cursor>` delta pull incl. tombstones; batch upsert
  by client UUID; append-only insert for `attempts` / `review_logs`.
- `/v1/generate`: persist the `source` (file → Storage, `storage_path` recorded) and the generated
  `cards`/`questions` under the user. `/v1/grade`: persist the `attempt` (incl. `feedback`).
- **Acceptance:** end-to-end pytest — create subject → generate (rows + file persisted) → pull
  delta → grade (attempt persisted) → soft-delete → tombstone appears on next pull. Cross-user
  isolation asserted.

  **Implemented as:** `app/repository.py` is the single owner-scoped data-access layer (ADR 0008
  §3) — it centralizes ownership scoping, parent-FK ownership checks (anti parent-stealing),
  upsert-by-client-UUID (id-squat → 409 on the global PK), and the soft-delete cascade that
  tombstones sync-table descendants. `app/sync.py` is a compound keyset cursor (`(updated_at|
  created_at, id)`) — required because a batch upsert shares one timestamp, so a bare-timestamp
  cursor would skip/loop. `app/routers.py` is a factory wiring all 8 resources uniformly through
  the repository; append-only `attempts`/`review_logs` get insert+read only. `app/storage.py`
  uploads source files to Supabase Storage (service-role REST, injectable for tests).
  `app/persist.py` persists generate (find-or-create subject → source+upload → cards → quiz+
  questions) and grade (append-only attempt via optional `question_id`). Tests:
  `backend/tests/` (cursor unit test runs without a DB; e2e + isolation + keyset-pagination need
  `TEST_DATABASE_URL`). **Deferred follow-up:** the FORCE-RLS / non-owner-role open question
  below is still open; revisit in Phase 4 prod hardening.

### Phase 4 — Pre-deploy hardening (carried O1 / M1 / prod) ✅ DONE (2026-06-18, ADR 0009)
- **O1 ✅:** per-caller rate limit on **all** `/v1/*` (router-level dependency; fixed per-minute
  window, Postgres `rate_limit_buckets`, per-user / per-IP for dev fallback; `429` + `Retry-After`).
  **Anthropic spend cap** — token-based, **daily (UTC), per-user *and* global**, summing the new
  `ai_usage_events` ledger; refuses with `429` **before** the Claude call when over budget.
  (Chose token-based over cost-based: exact from SDK usage, no price table. A monthly/cost ceiling
  can layer on the same ledger later.)
- **M1 ✅:** documented hard **reverse-proxy body cap** in front of the app-level cap —
  `backend/deploy/reverse-proxy.example.conf` (nginx/Caddy/Traefik) + `docs/SETUP.md`.
- **prod ✅:** `check_production_config` (in `app/config.py`, unit-tested) refuses to boot when
  `CRAM_ENV=prod` unless auth is configured, dev-fallback is off, `ANTHROPIC_API_KEY` + `DATABASE_URL`
  are set, and all three ceilings are `> 0`.
- **Acceptance — met:** rate limit returns `429` past threshold; spend cap blocks past budget without
  calling Claude; deploy checklist in `docs/SETUP.md`; prod boot fails fast on missing config. New
  table + migration `0003_ai_usage_and_rate_limit` (apply to Supabase like Phase 1). **24 tests green.**
- **Security review (2026-06-18) — 1 High found + fixed:** the spend cap was bypassable because usage
  was committed in the **same transaction** as the fallible attempt/deck persistence — a `/v1/grade`
  with a bogus `question_id` paid for Claude, then `422`'d on persist and rolled back the usage row
  (unmetered paid calls). **Fix:** commit the usage row on its own immediately after the paid call,
  before persistence (regression test `test_paid_call_is_metered_even_when_attempt_persist_fails`).
  No Critical; Info items (global-sum RLS dependency, dev-only XFF handling, residual pre-auth DoS,
  unbounded table growth) recorded in ADR 0009.
